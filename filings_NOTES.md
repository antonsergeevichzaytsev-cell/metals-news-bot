# filings-бот — где мы остановились

Состояние на **17.07.26, вечер пятницы**. Заметка живёт в репо нарочно: чаты удаляются, память у проектов раздельная, а репо читается откуда угодно.

## Что это

Второй бот в репо (первый — `digest.py`, новостной). Ловит первичные корпоративные релизы junior/mid miners и превращает их в повод написать: **что сломано + чем Антон закрывает**. Это труба с фильтром, НЕ агент: сам ничего не решает и не делает. **Себя не правит и править не будет** — см. «Почему нет самоулучшения» ниже.

Файлы: `filings.py`, `filings_sources.txt` (11 лент), `.github/workflows/filings.yml` (5 прогонов Пн–Пт), `state_filings.json`, `filings_history.json`.

## ГЛАВНОЕ: главная функция НЕ ПОДТВЕРЖДЕНА

За всю жизнь бот выдал **один** элемент, и тот бракованный. Всё, что залито после этого (гейт по стадии, контрастный пример, отсечка low, метки), **ни разу не проверялось живой моделью** — ноль элементов прошло через новый промпт. Тесты были на заглушке — проверена проводка, не решение.

Ничего не достраивать, пока не станет ясно, что гейт держит.

## Метки — как работает

Каждый хук приходит **отдельным сообщением**. Антон отвечает на него reply: `+` / `годится` / `беру` → good, `−` / `мимо` / `шум` → bad, любой другой текст → note (сохраняется дословно). Следующий прогон подбирает ответ через getUpdates и пишет в `filings_history.json` → `labels` (вердикт, сырой текст, компания, приоритет, стадия).

**Ограничения — знать обязательно:**

- **Telegram держит непрочитанные updates только 24 часа**, а бот ходит только Пн–Пт. Ответ, оставленный в субботу или воскресенье, **пропадёт**. Размечать — в будни.
- Метка есть только у того, что бот отправил. Зарезанное разметить нельзя — ложные отрицания видны только через флаг `orbit` в `skipped`.
- Токен общий с новостным ботом. `filings.py` — единственный, кто читает getUpdates; если когда-нибудь появится второй читатель или webhook — будут красть друг у друга обновления.

## Почему нет самоулучшения (решение 17.07, не пересматривать без цифр)

Антон просил «чтобы бот анализировал себя и улучшался». Отказано, три причины:

1. **Выборка.** 5-10 меток в неделю — любая правка промпта на таком объёме — подгонка под шум.
2. **Правило самого Антона** из его Программы: ИИ делает черновую аналитику, суждение — всегда за ним.
3. **Возвращает слепоту,** которую весь бот и лечит: самоправящийся фильтр может тихо съехать в «резать всё» при зелёном прогоне. Плюс ему нужен токен с правом писать в собственный репо — большая дыра ради нуля.

Метки ценны сами по себе: с ними промпт крутит человек, при Антоне, и каждая правка видна.

## РАЗБОР — что сделать в понедельник (18:00 MSK или позже)

Читать без токена, с CDN (кэш ~5 мин, бить cache-buster'ом):

- `https://raw.githubusercontent.com/antonsergeevichzaytsev-cell/metals-news-bot/main/state_filings.json`
- `https://raw.githubusercontent.com/antonsergeevichzaytsev-cell/metals-news-bot/main/filings_history.json`

**Считать пять вещей:**

1. **`state.last_run`** — воронка: `fresh` → `prefiltered_out` → `screened` → `skipped_by_model` → `kept`. Где режется больше всего. Если `cap_hit: true` — потолок 40 стал узким местом.
2. **`history.skipped`, флаг `orbit: true`** — **главный вопрос**: сколько монгольских/казахских релизов модель зарезала. Ноль — фильтр честный. Несколько — тугой. Смотреть сами заголовки, не только счёт.
3. **`history.labels`** — вердикты Антона. good/bad по приоритету и стадии: если 🔴 стабильно получают bad — кривой порог приоритета; если bad кучкуется на одной стадии — гейт там не держит. Читать `raw` у note — там живая причина.
4. **`history.items` со `stage`** — держит ли гейт. Стадия должна совпадать с хуком: стройка → контроль подрядчика, НЕ FEL. Запись без `stage` — старая, до фикса.
5. **`state.feed_health`** — все 11 должны быть `ok`.

Если есть хоть один 🔴 с нормальным хуком — **брать и писать письмо** (→ `cold-outreach`). Вот тогда бот окупится.

## Две известные дыры (не чинить до разбора)

1. **Бот не знает, кому Антон уже писал.** Выдаст «напиши в Erdene», не зная, что касание ушло. Перед любым письмом по хуку бота — свериться с `outreach_tracker` (каденция 4-7 дней, макс 2-3 касания).
2. **Одна компания = несколько релизов за неделю** → несколько хуков по одному активу. Шум это или нет — на одном элементе не видно.

## Что было сломано и почему (чтобы не наступить снова)

- **Не та лента.** Erdene вещает боевые релизы через GlobeNewswire; на Newsfile про него только шум от третьих лиц. **Одна лента = одна слепая зона.**
- **CV в промпте → модель выдаёт CV вместо ответа.** Первый же хук предлагал FEL проекту, который уже начал стройку. Лечится не «пиши конкретнее», а принуждением назвать стадию ДО хука.
- **Голый запрет обходится синонимом.** Запретил «предложить экспертизу» — получил «может потребоваться». Образец (WRONG/RIGHT) задаёт форму, запрет — нет.
- **Слепота хуже поломки.** Лента, умершая тихо, выглядела как лента без новостей: тишина + зелёный прогон. Теперь fetch называет причину, алерт по фронту.
- **Мой собственный промах (17.07):** заявил, что GlobeNewswire рубит раннер по IP (403). **Неверно** — все 11 лент `ok`. Диагноз был построен на кэшированном чтении CDN. Урок: сначала приборы, потом версии.

## Техническое

- Коммиты в репо — через Zapier GitHub-коннектор (`github_create_or_update_file`, `GitHubCLIAPI`). Для UPDATE нужен blob SHA — считается локально: `sha1(b"blob %d\0" % len(data) + data)`.
- **`.github/workflows/` коннектор НЕ может** (нет scope `workflow`, отдаёт "Not Found"). Только Антон вручную через веб.
- GitHub API без токена — 60 запросов/час на общий IP, часто выбит. Логи Actions без авторизации НЕ читаются — именно поэтому цифры прогона ложатся в `state.last_run`.
- Warning в Actions про Node.js 20 — косметика, висит на всех workflow репо, игнорить.
- Ключевые коммиты 17.07: `03018154` ленты GNW · `e0ea8cc` гейт по стадии · `a3d7daee` датчики лент · `a1a2228b` лог отказов + счётчики + потолок 40 · `6595a406` метки из ответов в Telegram.

## Стиль работы

Клод делает работу сам (пишет, тестит, коммитит), Антон реагирует. Никакого легворка/домашки. Каждая правка — сначала тест на копии, скачанной ОБРАТНО из репо, потом коммит. Промпты и код — EN, комментарии и UI — RU.
